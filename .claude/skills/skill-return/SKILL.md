# Skill Return

`playground-spawn` の対になるスキル。派生リポジトリで育ったものを playground に戻す。

これがないと playground は作った時点で凍りついて、実際に使われて改良されたスキルは
派生リポジトリに散らばったまま失われる。還流の経路を明示的に持つことで、
playground が「一度作って終わり」ではなく育つ資産になる。

## いつ使うか

- ユーザーが「これは playground に戻したい」と言ったとき
- 派生リポジトリでスキルを改良し、その改良が他のプロジェクトでも効きそうなとき
- 派生リポジトリで新しいスキルを作り、汎用性が高いと判断したとき
- 派生リポジトリでの作業が一段落したとき（棚卸しのタイミングとして自然）

自発的に「還流しませんか」と提案してよいが、**実行は必ず承認を得てから**。
playground への書き込みは CLAUDE.md で制限されており、還流はその例外ケースにあたる。

## Workflow

### Step 1: 還流候補の洗い出し

派生リポジトリの `.claude/SKILLS_ORIGIN.md` を読む。無い場合は Step 1b へ。

マニフェストの情報をもとに、以下の3種類に分類する。

**改変されたスキル** — マニフェストの「改変」列が `あり` のもの。加えて、列が `なし` でも
実際には変わっている可能性があるので、playground の該当ファイルと `diff` を取る。

```bash
diff -u <playground>/.claude/skills/<name>/SKILL.md .claude/skills/<name>/SKILL.md
```

**新規作成されたスキル** — マニフェストの「新規作成」表にあるもの。汎用性 `高` `中` が
候補、`低` は原則対象外（ユーザーが希望すれば含めてよい）。

**マニフェストに無いスキル** — `.claude/skills/` にあってマニフェストのどちらの表にも
無いもの。記録漏れの可能性が高いので、新規作成として扱いユーザーに確認する。

#### Step 1b: マニフェストが無い場合

`.claude/skills/` の各フォルダを playground の同名フォルダと突き合わせる。
playground に無いものは新規、あるものは `diff` を取る。マニフェストがある場合より
手間はかかるが、判定自体は可能。

この場合、還流と併せて `.claude/SKILLS_ORIGIN.md` を新規作成することを提案する。
次回以降が楽になる。

### Step 2: 候補の提示

以下の形式で提示し、還流するものを選んでもらう。**この段階では何も書き込まない。**

```
## 還流候補

### 改変されたスキル: N件

#### 1. skill-creator
- 変更内容: ○○のセクションを追加、descriptionを△△に変更
- 他プロジェクトでも有効か: はい（理由）
- 衝突リスク: playground 側は持ち出し後に変更なし → 単純上書きで安全

### 新規作成されたスキル: M件

#### 2. api-mock-server
- 用途: ...
- 汎用性: 中（REST API を扱うプロジェクトなら有効）
- description案: ...

還流するものの番号を教えてください。
```

**衝突リスクの判定** は重要。マニフェストの持ち出し元 hash 以降に playground 側で
同じファイルが変更されていないかを確認する。

```bash
git -C <playground> log --oneline <origin-hash>..HEAD -- .claude/skills/<name>/
```

出力があれば playground 側も変わっているので、単純上書きは危険。3-way の差分を
ユーザーに見せて判断を仰ぐ。

### Step 3: 品質チェック

承認された候補について、playground に入れる前に `skill-optimizer` を通す。
特に確認すべきは以下。

- description が既存スキルと発火条件で衝突していないか
- 派生リポジトリ固有の前提（特定のパス、特定の技術スタック）が本文に残っていないか
- 汎用化のために書き換えるべき箇所はないか

派生リポジトリ固有の記述が残っていると、playground に入れた後に他のプロジェクトで
持ち出したときに混乱の元になる。ここで一般化しておく。

### Step 4: playground への反映

playground をクローンまたは pull してから作業する。

```bash
git -C <playground> pull
```

承認されたスキルを配置する。

```bash
cp -r .claude/skills/<name> <playground>/.claude/skills/
```

改変の還流で playground 側も変更されていた場合は、上書きせずマージする。
どう統合したかをコミットメッセージに残す。

### Step 5: 記録

`memory/CHANGELOG.md` に1行追記する。

```
- YYYY-MM-DD: 還流。<repo-name> から skill +N（<names>）。
```

還流は棚卸しの一種なので、`memory-sync` と同じ CHANGELOG を使う。
これにより playground の変更履歴が一箇所に集まる。

### Step 6: コミットと push

```bash
git -C <playground> add -A
git -C <playground> commit -m "return: <repo-name> から <names> を還流"
git -C <playground> push
```

これは CLAUDE.md の「更新してよい例外ケース」に該当するため実行してよい。
ただし既存ファイルの上書きを含む場合は、push 前にもう一度確認する。

### Step 7: 派生リポジトリ側のマニフェスト更新

還流が完了したら、派生リポジトリの `.claude/SKILLS_ORIGIN.md` を更新する。
還流済みのスキルは「改変」列を `なし（YYYY-MM-DD に還流済み）` に戻し、
新規作成表から持ち出し表に移す。

これをやらないと次回の還流で同じものが二重に候補に上がる。

## 判断基準: 還流すべきか

**還流すべき**

- 手順・判断基準が一般化できる（特定のプロジェクトに依存しない）
- 同じ種類のタスクが他でも発生しうる
- 試行錯誤の末に確立したもので、失うと再現に時間がかかる

**還流すべきでない**

- そのプロジェクトのドメイン知識に強く依存する
- 一度きりの作業を記録しただけ
- 既存スキルの機能とほぼ重複する（→ 既存スキルへの追記として扱う）

判断に迷ったら還流する側に倒してよい。playground は素材置き場なので、
使わない素材があっても害は小さい。逆に失われた知見は戻らない。

## このスキルがやらないこと

- 新リポジトリの作成・スキルの持ち出し（`playground-spawn` の担当）
- claude.ai 側との同期（`memory-sync` の担当）
- スキルの品質診断そのもの（`skill-optimizer` に委譲する）
- Skill 候補の発掘（`skill-harvester` の担当。あれは会話やCLAUDE.mdから候補を探すもので、
  こちらは既にスキルの形になっているものを移送する）
