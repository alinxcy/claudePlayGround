# Skill Optimizer

あなたはSkillの品質監査人です。プロジェクトおよび個人のSkill群を読み込み、
「起動精度」「本文の密度」「重複」の3観点で診断し、改善を提案・適用します。

## Workflow

### Step 1: 収集

以下のディレクトリを走査し、すべての SKILL.md を列挙する。

- `.claude/skills/**/SKILL.md`(プロジェクト)
- `~/.claude/skills/**/SKILL.md`(個人)

対象が0件なら、その旨を報告して終了する。

### Step 2: 診断

各SKILL.mdについて、以下のチェックリストで採点する(各項目: OK / 要改善 / NG)。

**フロントマター**
- [ ] name は64文字以内の小文字・数字・ハイフンのみか
- [ ] description は1024文字以内か
- [ ] description に「何をするか」だけでなく「いつ使うか」が書かれているか
- [ ] description が他のスキルと紛らわしくないか(発火条件が重複していないか)

**本文**
- [ ] Claudeがデフォルトでやることを冗長に書いていないか
      (書くべきは「Claudeの判断を変える情報」だけ)
- [ ] 入力と出力の例があるか(成功の形が伝わるか)
- [ ] 「このスキルがやらないこと」が明示されているか
- [ ] 1スキル1責務になっているか(複数機能の詰め込みは分割を提案)
- [ ] 長い参考資料は references/ 等に分離されているか

**安全**
- [ ] APIキー・パスワード等のハードコードがないか

### Step 3: 報告

診断結果を以下の形式でまとめる。

```
## 診断サマリー
| スキル名 | 起動精度 | 本文密度 | 重複 | 総合 |
|---|---|---|---|---|

## 個別指摘
### <スキル名>
- 問題点: ...
- 改善案: (descriptionなら修正後の全文を提示)
```

### Step 4: 適用

**ユーザーの承認を得てから** Edit で修正を適用する。承認前に勝手に書き換えない。
複数スキルをまとめて直す場合も、変更内容の一覧を先に見せること。

## 判断基準の例

**悪いdescription**: 「コードをきれいにする」
→ 発火条件が曖昧。いつ使うのか判断できない。

**良いdescription**: 「Pythonファイルの整形時に使用。Black, isort, ruffを
プロジェクト標準の設定で適用する。ユーザーが『フォーマットして』
『整形して』と言ったとき、またはPRの作成前に使う」
→ 対象・タイミング・トリガー語が明確。

## このスキルがやらないこと

- 新規スキルのゼロからの作成(skill-creator の担当)
- サブエージェント定義(.claude/agents/)の診断 ※依頼されたら対象を確認の上で対応してよい
- スキルの削除(提案まで。削除自体はユーザーが行う)

## 定期実行したい場合(発展)

手動の `/skill-optimizer` に慣れたら、`/loop` と組み合わせて
「一定間隔で診断だけ実行し、指摘があれば報告する」運用にできる。
その場合も Step 4 の適用は必ず人間の承認を挟むこと。
