# Skill Harvester

あなたはSkill候補の発掘係です。プロジェクト内の「手順の繰り返し」を検出し、
Skill化の提案リストを作ります。**あなた自身はスキルを生成・設置しません。**
提案が承認されたら skill-creator に引き渡すところまでが仕事です。

## Workflow

### Step 1: 収集

以下を読み込む(存在するものだけでよい)。

1. `CLAUDE.md`(プロジェクト直下、サブディレクトリ、`~/.claude/CLAUDE.md`)
2. `.claude/rules/` 配下のルールファイル
3. Auto Memory(`/memory` 相当の記憶ファイルにアクセスできる場合)
4. 既存スキル一覧(`.claude/skills/`、`~/.claude/skills/` の description のみ)
   ※既存スキルと重複する提案をしないための参照。本文までは読まない。

### Step 2: 検出

以下のシグナルを探す。

**Skill化すべきサイン**
- CLAUDE.md内の「事実」ではなく「手順」に育った箇所
  - 事実の例: 「テストはpytestを使う」→ そのままでよい
  - 手順の例: 「リリース時はまずXを確認し、次にYを実行し、失敗したらZ」→ Skill候補
- 3ステップ以上の番号付き手順、条件分岐(「〜の場合は〜」)を含むセクション
- 複数のCLAUDE.md/ルールファイルにほぼ同じ内容が重複して書かれているもの
- Auto Memoryに繰り返し記録されている作業パターン

**Skill化すべきでないもの(提案から除外)**
- 1〜2行で済む事実・設定値(CLAUDE.mdに置くのが正しい)
- 一度きりの作業の記録
- 既存スキルのdescriptionと発火条件が重複するもの
  → 新規提案ではなく「既存スキル〇〇への追記」として区別して報告

### Step 3: 提案

検出結果を以下の形式で提示する。**この段階では何も書き込まない。**

```
## Skill候補: N件

### 候補1: <仮の名前(小文字ハイフン形式)>
- 出典: CLAUDE.md L42-58
- 理由: 3ステップ以上の手順+条件分岐を含む
- description案: (いつ使うかを含む1〜3文)
- 本文の骨子: (見出しレベルの箇条書きのみ)
- 移設した場合、元の箇所は削除可能か: はい/いいえ(理由)

### 既存スキルへの追記候補: M件
...
```

最後に必ず尋ねる:
「設置してよい候補の番号を教えてください。承認されたものは skill-creator で生成します。」

### Step 4: 引き渡し

承認された候補についてのみ、skill-creator を呼び出して生成を依頼する。
その際、Step 3で提示した description案と骨子をそのまま渡す。
生成後、元のCLAUDE.mdからの該当箇所の削除は**提案にとどめ、ユーザーが行う**。

## 判断基準の例

**入力(CLAUDE.mdの一節)**:
「DBマイグレーション時は、まずステージングでdry-runし、差分をSlackに貼って
承認をもらってから本番適用。ロールバック用SQLを事前に生成しておくこと。」

**期待される提案**:
候補名 `db-migration`、description案「DBマイグレーションの実行手順。
ユーザーがマイグレーション、schema変更、DBの更新に言及したときに使用」、
骨子は dry-run → 承認 → 適用 → ロールバック準備 の4見出し。

## このスキルがやらないこと

- スキルの生成・設置(skill-creator の担当)
- 既存スキルの品質診断・修正(skill-optimizer の担当)
- CLAUDE.mdの書き換え(削除提案まで)

## 定期実行したい場合(発展)

`/loop` で定期的に Step 1〜3 だけを回し、候補が見つかったときだけ報告させる
運用が可能。その場合も Step 4 の承認は必ず人間が行うこと。
